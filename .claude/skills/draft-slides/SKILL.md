# draft-slides スキル

論文 PDF から，ゼミ・輪講用スライドの**構成 draft** を markdown で生成するスキル．PowerPoint 自動生成ではなく，**手作業でスライドを作る際の参考資料**を作る．

---

## 前提

- 対象 PDF が `raw/papers/` に置かれていること
- `slidegen/drafts/` ディレクトリが存在すること
- 出力先は `slidegen/drafts/<slug>.md`．`<slug>` は PDF ファイル名の拡張子除去部 (例: `raw/papers/author2024title.pdf` → `slidegen/drafts/title.md`)．**既存のファイルがあれば上書き確認する**

## 参照する情報源

処理前に以下を読む:

- **PDF 本体** (Read ツールで全ページ，10 ページ刻み)
- **`my-research/current.md`** — 所感スライドで本研究との関連を書くため
- **`wiki/papers/<slug>.md`** — 存在すれば事実ベースの要約を下地として利用 (ingest-paper で既に作成済みの場合)

## 出力 markdown の構造

### Frontmatter

```yaml
---
title: "<論文の正式タイトル>"
authors: "<第一著者ほか>"
venue: "<会議名 年>"
presenter: "<発表者>"
date: "<YYYY-MM-DD (今日の日付)>"
source_pdf: "raw/papers/<filename>.pdf"
---
```

### スライド 1 枚の基本構造

```markdown
## <スライドタイトル>
<!-- layout: 本文A -->
<!-- figure: <図の作り方や貼るべき論文図の指示> -->

本文 (見出し風プレーン段落と箇条書きの組み合わせ)

- 箇条書き 1
- 箇条書き 2

*<用語>: <軽い補足説明>

> [<N>] <Author et al. Year. Title. Venue.>
```

区切りは `---` (horizontal rule)．

### スライドタイトルの 2 段構造

複数枚にまたがる章 (RQ 展開・考察・技術要素の段階説明など) では **2 段タイトル**を使う．**単独枚で完結する章は 1 段で可** (データセット / まとめ / 所感など)．

**「章」の定義**: 同じ上段タイトルを 2 枚以上で繰り返して使う構造．以下のいずれかに該当すれば「章」扱い:

- RQ を複数枚に分割する (例: RQ1 を「評価方法」「結果 A」「結果 B」の 3 枚に展開)
- 考察を複数枚に分割する (例: 「なぜ X は Y なのか」を 2 枚で深掘り)
- 技術要素を段階ハイライトで複数枚に展開する
- 評価が RQ 型でなくても，測定軸が異なる 2 枚以上を束ねる上位概念がある場合 (例: 「性能比較」章 / 「ケーススタディ」章)

**「章」にしない場合** (1 段タイトルで可):

- 1 枚で完結する内容．上段を次のスライドで繰り返さない場合
- 独立した固有名を各スライドが持つ場合 (例: `case study: Herbie` と `case study: Spores` が別々の論点を扱い，共通の上段で括る意味が薄い → 各々 `## case study: Herbie` / `## case study: Spores` の 1 段タイトルで OK)
- 逆に，複数 case study を共通の上段で束ねたい場合は 2 段化も可 (例: `## case study` / `### Herbie` を 3 連続)．論文の流れと聴衆の追いやすさで選ぶ

```markdown
## RQ1: COSMosFLと従来研究の効果の違いを評価
### パフォーマンス
```

- 上段 (`##`): 大きい問い．章全体を指す短文
- 下段 (`###`): このスライドでの focus

上段は複数枚にわたって同じ文を使い，下段だけ変えるのが定石．聴衆が「今どの章のどの側面を見ているか」を失わない．**1 枚だけで終わる章 (= 同じ上段を繰り返さない) では 2 段構造は不要**．

## 全体構成 (20 スライド目安)

以下の順に組み立てる．論文の性質により増減あり．**推奨範囲は 18-22 枚**．25 枚を超える場合は以下を検討:

- 補足資料に回せる内容がないか (具体例・実装詳細等)
- 同一 RQ の分割が多すぎないか (3 枚まで)
- 技術要素の段階ハイライトが細かすぎないか

| # | スライド | 必須/任意 | 内容 |
|---|---|---|---|
| 1 | 表紙 | 必須 | 論文タイトル・著者・会議・発表者・日付 |
| 2 | 背景 | 必須 | 分野の前提，基本概念の導入．SE 系聴衆に馴染みない用語は必ず説明 |
| 3 | 既存の問題 | 必須 | 論文が解こうとしている問題．「だからこそ本論文が必要」の論立て |
| 4 | 目的 / 提案手法 | 必須 | 目的と提案手法の全体像を 1 枚に統合 (空きが足りなければ分割) |
| 5 | 前提知識 (道具) | **任意** | SMT ソルバ / 形式検証器など，複数の後続スライドで**横断的**に使う道具がある場合のみ配置 |
| 6〜N | 技術要素 | 必須 | 論文の中核技術を 1 技術 = **1〜2 枚**で．大きなステップを分ける時は「段階ハイライト図」を使う (図の方針節) |
| N+1 | データセット | 必須 | 評価に使うデータソースの紹介と選定基準．複数データセットがあれば対比表 |
| N+2 | RQ 一覧 or 評価の全体像 | 必須 | RQ 型の論文は「RQ 一覧」．RQ 型でない論文 (ツール論文・ベンチマーク論文) は「評価の全体像」で評価軸を列挙 |
| N+3〜 | RQ ごとの結果 / 各評価の結果 | 必須 | **1 軸 = 1〜3 枚** (分割基準は評価スライド節) |
| ... | 考察 | **任意** | 意外な結果・反直感的な結果を深掘り．RQ の後・まとめの前 |
| 最後 -3 | 貢献 | 必須 | 貢献 4-5 項目．**まとめ の直前に配置** (冒頭ではない) |
| 最後 -2 | まとめ・限界 | 必須 | 結論と制限を同じスライドで 2 ブロック |
| 最後 -1 | 所感 | 必須 | 発表者個人の意見．本研究 (my-research/current.md) との関連，数値への批判，応用可能性など |
| 補足 | 補足資料 | 推奨 | 質疑対応用．**2 枚以上**．タイトルに「補足資料:」プレフィックス |

**必ず守る**:

- **貢献は冒頭に置かない**．まとめの直前に独立スライド化
- **所感スライドを最終スライドに含める**．ここで my-research との接続を書く

## 評価スライドの構造ルール

論文の評価構造 (RQ 型 / ツール論文型 / その他) に応じて適用する．

### RQ 型の論文 (RQ1-RQ4 など明示的な RQ がある)

**分割基準 (1 RQ = 1〜3 スライド)**:

| 測定指標の数 / 視点の数 | スライド数 | 構成 |
|---|---|---|
| 数値のみ (図表なし) | 1 枚 | 実験構成 + 結果 を同一スライド |
| 1 つの図/表あり | 2 枚 | 構成 / 結果 (図) |
| 2 つ以上の図 (例: Fig.8+Fig.9) | 2〜3 枚 | 構成 / 結果 1 / 結果 2 |
| 別視点分析 (例: パフォーマンス vs 堅牢性) | 2〜3 枚 | 視点ごとに独立．タイトル下段で区別 |

構成順: `データセット → RQ 一覧 → RQ ごとの結果`．

### RQ 型でない論文 (ツール論文，ベンチマーク論文，ケーススタディ型)

「RQ 一覧」の代わりに「**評価の全体像**」スライドを置く:

- 論文が立てている評価軸 (性能比較 / スケーラビリティ / ケーススタディ等) を箇条書きで列挙
- 各評価軸に対して 1〜3 枚のスライドを配置 (分割基準は RQ 型と同じ表を適用)

**記述様式の例**:

```markdown
## 評価の全体像
<!-- layout: 本文A -->

本論文は 3 つの軸で評価する:

- 軸 A: 速度比較 (§3.4 / Fig.6, Fig.7)
- 軸 B: ケーススタディ (§6 / Herbie, Spores, Szalinski の 3 つを個別分析)
- 軸 C: スケーラビリティ (§3.2 / Fig.8)
```

各軸の節番号・該当図は**併記を推奨** (聴衆がどこを見ればいいかすぐ分かる)．

タイトル 2 段構造の「章」= 同じ評価軸を束ねる上位概念 (例: 「性能比較」章 の下に「速度」「スケール性」2 枚).

### 重複回避 (RQ 型・非 RQ 型共通)

- 評価が複数枚に分かれる場合，**実験構成は初回のみ書く**．2 枚目以降は結果ブロックのみ
- データセットが全評価で共通なら**データセットスライドのみ**に書き，各評価では言及不要
- 複数構成比較 (アブレーション・他手法との対比等) も初回のみ．2 枚目以降は結果のみ

### 対応フッター (必須)

**対象スライド**: 論文の特定の図表・節を根拠として示すスライド．具体的には:

- RQ スライド (評価方法・結果ともに)
- 考察スライド (論文の別節を引いて深掘りする場合)
- RQ 型でない論文の各評価スライド (性能比較・ケーススタディ等)

**対象外** (対応フッター不要):

- 表紙・背景・問題提起・目的/提案手法スライド (論文全体の要約なので特定節を指さない)
- 技術要素スライド (論文の構造をなぞっているので特定図を指さない，ただし「この規則を §4.2 の Fig.4 に合わせた」など特定図を引用する場合は付ける)
- RQ 一覧 / 評価の全体像 (具体的な図表がない)
- 貢献・まとめ・所感・補足資料

書式:
```
対応: 論文 §X / Fig.Y or Table Z (図表の役割を短く)
```

**配置場所**: タイトル行 (`##` / `###`) と `<!-- layout: -->` / `<!-- figure: -->` コメントの**直後**，本文の**最初の行**．

例:
```markdown
## RQ2: 発火頻度 (23.7× 増)
### 結果
<!-- layout: 本文B -->
<!-- figure: 右半分に PDF p.18 Fig.7 のグラフ -->

対応: 論文 §8.3 / Fig.7 (発火数の順位比較グラフ)

分析手法
- ...
```

## 考察スライドの構造 (任意)

論文の結果に**意外な点**・**反直感的な点**があれば，RQ の後・まとめの前に考察スライドを 1〜3 枚挟む．

### 配置判断

- 「重み最適化に効果がなかった」「予想と逆の結果」など，因果の掘り下げが必要な場合: 追加
- 結果が期待通りで特に深掘りがない場合: 省略

### 構成

```markdown
## 考察
### なぜ <観察された現象> なのか

対応: 論文 §X (あれば)

評価方法 (追加分析の手順)
- ...

結果 (観察されたこと + 図)
- ...

理由 (観察を裏付ける仮説 + 表/例)
- ...

> 結論メッセージ (1 行で要約)
```

最下段にメッセージボックス (引用構文 `>` で擬似的に表現) を置くと，考察の要点が一目で伝わる．

**記法の注意**: 上のテンプレで `<観察された現象>` のように `<...>` で囲まれた部分は**プレースホルダー** (発表者が論文に合わせて埋める箇所)．`<`, `>` の記号ごと削除して具体テキストに置き換える．例: `### なぜ 重み最適化は精度に効かないのか`．draft 生成時にこれらを `<...>` のまま残してはいけない (必ず論文内容に即した具体文を書く)．

## Style ルール (必須)

### 1. レイアウト使い分け

- **表紙**: 論文タイトル・著者・会議・発表者・日付の専用レイアウト (必ず 1 枚目)
- **本文A**: 1 スライド = 1 メッセージ．単一主張を大きく見せたいとき
- **本文B**: 2 つ以上のブロック (目的+提案手法，実験構成+結果，など) を 1 枚に載せたいとき
- **本文C**: 用途未確定．使わなくてよい
- **補足**: 質疑対応用の補足資料スライド．本編とは区別される扱い

構造的には本文A と本文B は同じだが**意味的**に使い分ける．1 スライド 1 メッセージ原則を優先し，迷ったら本文A を選ぶ．補足スライドは本編の layout (A/B) ではなく `<!-- layout: 補足 -->` を使う．

### 2. 文字ルール (非常に重要)

- **太字 `**...**` は使わない**．MS PGothic の太字が野暮ったく見えるため．見出し感は行の独立と level 0 で出す
- **全角読点は `，`** (`、` ではなく)
- **文末句読点なし** (`.` も `。` も使わない)．箇条書き・見出しは体言止めか無終止
- **本文中は `peephole`, `superoptimizer` 等を小文字**で統一 (見出し・タイトルは語頭大文字 OK)
- 全角括弧 `（）` は使わず半角 `()` 統一
- 矢印: 変形・結論は `→`，論理含意 (LHS ⇒ RHS) は `⇒` で意味で使い分け

### 3. レベル階層 (Level 0 vs 1)

- **Level 0** (プレーン段落): 小見出し．箇条書きの親項目．例: `peephole 最適化の特徴`
- **Level 1** (`-` 箇条書き): 具体的な bullet

Markdown 上は `- ` 付きか否かで区別する (テンプレの layout が level ごとに bullet char を自動付与する)．

### 4. 聴衆前提と用語説明

- 聴衆は **SE (ソフトウェア工学) 系研究室**．論文に出てくる固有名 (システム名・先行研究名) や分野固有の専門用語は**事前知識としない**
- 初出の専門用語は以下のいずれかで説明:
  - 日本語に書き換える (標準訳が定着している場合)
  - 英語のまま使い，**同じスライド内に短い括弧書きの説明**を付ける (例: `<システム名> (<対象領域> 向けの <一言での役割説明>)`)
  - 軽い補足は `*<用語>: <短い説明>` footnote (1 スライド 2 件まで)
- 訳が不自然な語 (定訳が定着していない or 訳語が逆に分かりにくい) は英語のまま使う
- **要素技術で複数スライドに登場する道具は「前提知識」スライド化可能** (全体構成の slide 5)．単発なら inline 説明

**英単語の扱いの原則 (3 階層)**:

判定は次の順に当てる．上位で該当したら下位には進まない．

| Level | 扱い | 対象 | 例 |
|---|---|---|---|
| **L1** | 英語のまま (そのまま) | 固有名詞・確立略語．訳しても逆に分かりにくくなるもの | システム名・ツール名・データセット名 (固有名詞)，`SMT`, `IR`, `RQ`, `SE`, `GNN`, `LLM`, `API`, `CPU`, `GPU` |
| **L2** | 併記: `日本語 (English)` or `English (日本語)` | 分野用語で定訳が未定着 or 両表記が混在．論文・スライド中で 2 回目以降は片側だけでも可 | 分野固有の概念語 (定訳が定まっていない技術用語) — 例: `<英語表記> (<日本語訳>)` の形で初出時に併記 |
| **L3** | 日本語化 (英語表記を消す) | 日常的に日本語で通じる一般用語．英語で書くと浮く | `issue tracker` → `不具合報告トラッカー`, `synthesis` → `合成`, `ungeneralized` → `未一般化`, `ablation` → `要素削除実験`, `pair` → `対`, `route` → `経路`, `before/after` → `変換前/変換後`, `LHS/RHS` → `左辺/右辺` |

**判定の迷い方**:

- L1 か L2 か迷う (例: `SMT solver`) → 略語単独 (`SMT`) は L1，語としての塊 (`SMT solver`) は L2 で `SMT ソルバ` と扱う
- L2 か L3 か迷う → 聴衆 (SE 系) が未修得の可能性が残るなら L2 (併記)．日常語感覚で通じるなら L3 (日本語化)
- 1 スライド内で同語が 3 回以上出るなら，初出のみ L2 で併記し，以降は日本語側 or 英語側どちらかに統一

### 5. 図の方針

#### 論文図の取捨選択

- **論文の Fig が直感的でない (抽象的・数式主体) 場合**の代替手段 (順に優先):
  1. **表で代替**: 論文に running example の評価表 (step-by-step 表) があればそちらの方が情報密度が高い
  2. **自作デフォルメ図**: PowerPoint 標準図形 (矩形・矢印・吹き出し) で簡略化して描く
  3. 論文図をそのまま: 最後の選択肢

- **Running example table** (論文に用意されている場合): 提案手法スライド (p.4) に抜粋を埋め込んでアプローチ全体像と連動させる

- **非自明な列の読み方** (例: `Valid?` 列の ✓/✗): 本文に読み方を必ず書く．「途中 ✗ が出て最後に ✓ に戻る」のような動的な読み方は特に

#### pipeline locator (全体図ミニチュア)

- **技術要素スライド**でアーキ全体が 3 つ以上のステップに分かれる場合，右上に **pipeline locator** (全体図の縮小 + 該当ステップのみ色付き，他はグレースケール) を置くと視線誘導が楽になる
- 2 ステップ以下なら locator 不要
- **RQ スライド・考察スライド**でも、アーキ図のどの部分に関する話か示したい場合は locator を流用可能 (任意)

#### 段階ハイライト図 (複雑図の分解)

複雑なアーキ図・パイプライン図を 1 枚で説明するのが無理なときは，**同じ図を複数枚連続で使い，各枚でハイライト位置を進める**:

- 1 枚目: 最初の Step のみ色付き，他はグレーアウト + 該当部分の吹き出し説明
- 2 枚目: 2 番目の Step を追加色付き + その部分の吹き出し説明
- ... (Step 数だけ続ける)
- 最終枚: 全 Step 色付き (完成形)

pipeline locator とは逆方向の働き: locator は俯瞰，段階ハイライトは拡大．併用可能．

#### 図指示の書き方

図指示は `<!-- figure: ... -->` コメントに**何をどう描くか具体的に**書く．pptx 手作業時の指示書として機能させる．

### 6. グラフの凡例対応

グラフを貼るスライドでは**本文中に凡例対応**を必ず書く:

```
- 赤 = 提案手法
- 青 = 従来手法 (ベースライン)
- 淡赤 = 提案手法 (補正・組合せ条件)
```

縦軸の解釈も書く:

- `Percent change (-ve is better)` → 「負が改善」と日本語でも明記
- 対数スケール / 線形スケール の違いがあれば明記

### 7. 引用ルール

- 元論文で引用されているもので**自分でも重要と判断したもの**を，**初出スライド**でスライド末尾にブロッククォート `> [N] Author et al. Year. Title. Venue.` で追加
- **ユニーク引用**: 同じ引用を別スライドで重複させない
- 引用番号 `[N]` は本文中の参照タグと一致させる

### 8. まとめ・所感スライドの構造

#### まとめ

簡素な箇条書きで可．以下の 3 段構造は**推奨だが必須ではない**:

```
目的: (1 行)
手法: (箇条書き 2〜4 点)
結果: (1 行．定量値を含む)
```

**推奨する時**: 論文に複数の手法要素がある場合．**省略して可**: 手法が単純な場合．

#### 所感

所感は 4 カテゴリのいずれかに属する内容で 3〜4 項目書く:

1. **自分の研究との接続** (最重要): my-research/current.md の方針とこの論文の関係
2. **論文への批評**: 測定方法・比較対象の適切さ・バイアスへの突っ込み
3. **応用可能性**: 他分野・他言語への転用の芽
4. **聞きたかったこと**: 論文が触れなかった観点・続編への期待

**避ける**: 「分野違い読むのしんどいね」のような発表者の独り言．聴衆に資する観点のみ．

### 9. 補足資料 (質疑対応用)

- **2 枚以上**推奨．1 枚では不十分．候補:
  - ベンチマーク内訳 / データセット詳細テーブル
  - 具体的なプロンプト例 (LLM 系論文の場合)
  - 論文の表/図の完全版 (本編は抜粋にしている場合)
  - 失敗ケースの詳細一覧
  - 計算環境の詳細
- タイトルに「補足資料:」プレフィックスを付けて本編と区別

### 10. 数字・記法

**数字対比**:
- ❌ `従来 6,158(/ 26,162) vs 提案 196 (/196)`
- ✓ `従来 6,158 件 (全 26,162 件中) vs 提案 196 件 (全 196 件中)`

**倍率**:
- ❌ `23.7×` と単独
- ✓ `23.7 倍 (提案 442,860 回 / 従来 18,695 回)`

**単位の日本語化**:
- `10 分 timeout` → `10 分のタイムアウト`
- `GB 単位` → keep

### 11. 構成順序の特殊ルール

- **貢献**は論文紹介の**冒頭ではなく，「まとめ」の直前**に独立スライドで配置．学会口頭発表型ではなく，筋道を追った後に貢献を sum up する輪講型の構成
- **目的と提案手法は統合 or 分割を空きスペースで判断**:
  - 提案手法に大きな図 (pipeline diagram, architecture 図) が入るなら**分割**
  - テキストのみで済むなら**統合**を第一候補

## 手順

1. **引数確認**: PDF パスが `raw/papers/<filename>.pdf` 形式か確認
2. **slug 決定**: `<filename>` (拡張子除く) を slug とする．出力先 `slidegen/drafts/<slug>.md` が既存なら上書き確認
3. **PDF 読み込み**: Read ツールで全ページ読む (10 ページ刻み推奨)．論文の章構造 (Abstract, Introduction, Motivation, Approach/Method, Evaluation, Discussion, Related Work, Conclusion) を把握
4. **補助情報読み込み**:
   - `my-research/current.md` を読む (所感スライドのため)
   - `wiki/papers/<slug>.md` が存在すれば読む (要約の下地)
5. **構成決定**:
   - 論文の性質を見て 20 スライド前後の構成を決める
   - 技術要素の数が多ければ増，少なければ減
   - RQ ごとに 1〜3 枚を測定軸の数で決定
   - 考察スライドの要否を判断
6. **draft 生成**: 上記 Style ルールと出力フォーマットに厳密に従って markdown を組み立てる
7. **保存**: `slidegen/drafts/<slug>.md` に書き出す
8. **報告**: 生成したスライド数と layout 内訳 (表紙 1 / 本文A X / 本文B Y / 補足 Z) を報告．気になった論点があれば短く共有

---

## 構成の目安

過去の成功例から抽出した目安．以下を踏襲しつつ，**論文の性質に応じて構成を調整する** (硬直的なテンプレートではない):

- 20 スライド前後 (表紙 1 / 本文A 多め / 本文B 多め / 補足 1)
- 前提となるシステム名・概念は提案手法スライド内で inline 説明 + 別途「前提知識」スライドで主要概念を導入
- 技術要素スライド (5〜10 枚) に pipeline locator 指示
- 所感スライド (最終) で **my-research/current.md** の関連節と論文の対応を記述

## 注意事項

- 本 skill は **markdown draft 生成のみ**．PowerPoint の pptx 生成・Marp 変換は行わない (slidegen/ にそれらのツールは置かない)
- 所感は発表者の視点が入るので，**my-research/current.md を読んでも「こう書け」とは一方的に決めず**，発表者が差し替え前提の下書きとして提示する．プレースホルダー `<発表者記入>` を残してもよい
- 論文が PDF 抽出で読みづらい場合 (OCR 不完全など) は読めた範囲で draft を作り，**読めなかった章を明記**して報告
- 図の具体的な配置位置 (cm 単位の座標等) までは指示しない．`右上 locator` `右半分にスクショ` 程度の粒度で OK
