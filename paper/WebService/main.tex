\section{Discord}

\subsection{サービス概要}
\begin{itemize}
  \item \textbf{名称：} Discord
  \item \textbf{トップページURL：} \url{https://discord.com}
  \item \textbf{概要：} ゲーマー向けのトークアプリである．サーバをたて，テキストチャットやボイスチャットを，PCやスマートフォンから利用できる．最近では，ゲーム以外のコミュニティも多く利用している．
\end{itemize}

\subsection{選んだ理由}
Discordは，オンラインコミュニケーションのためのプラットフォームであり，特にゲーマーの間で広く利用されている．したがって，低レイテンシかつリアルタイム性が求められるサービスである．そこで，このようなサービスにおいてどのような技術的工夫がなされているかに興味を持ち，本サービスを選んだ．

\subsection{定量情報}
Discordの会社情報\cite{discord-company}によると，DAUは9,000万人以上である．また，MAUは2億人以上であり，そのうち90\%以上のユーザがゲームをプレイしており，1時間以内にゲームを開始するユーザは最大40\%である\cite{discord-ceo-2025}．現在では，これら2億人以上の月間アクティブユーザが数千ものタイトルで毎月20億時間ものゲームをプレイしている\cite{discord-blog-world-plays}．

\subsection{サービスの構成}

本節では，Discordが大規模化に伴って直面した課題と，それを解決した技術的工夫を，インフラ全体・CDN・メッセージ検索基盤・データベース・音声通信基盤の5つの観点に分けて述べる．

\subsubsection{インフラ全体と言語選定}

Discordのインフラ全体としては，Google Cloud Platformを利用している\cite{discord-cloudflare}．Discordは2015年の創業当時からElixirを使用しており，現在もインフラのバックボーンとして機能している．リアルタイム通信においてErlang VMは最高のツールであるため，Elixirを選択しており，毎秒2,600万件以上のメッセージを処理している\cite{discord-elixir-scale}．しかし，全体の構成としてはハイブリッドであり，APIはPython，リアルタイム通信はElixir，メディアストリーミングはC++，一部のデータサービスはRustで実装されている．

\subsubsection{CDNとDDoS対策：Cloudflare}

一方で，ターゲット層がゲーマーであるということも相まって，嫌がらせ目的でのDDoS攻撃を受けることが多い．また，従来と同様の負荷分散では，年間数十万ドルのコストがかかる見込みであり，運用の負担も限界に達していた．そこで，Cloudflare CDNを導入することで，世界中のデータセンターでコンテンツをキャッシュし，さらに，DDoS攻撃を自動的に検知し，攻撃トラフィックをブロックすることで，コスト削減とセキュリティの向上を実現した\cite{discord-cloudflare}．これによって，わずか1年でユーザ規模が240万人（前年比9000\%）へと急成長を遂げた．また，毎秒200万もの攻撃をCloudflareがブロックすることで，攻撃トラフィックを減らし，ユーザへの影響を最小限に抑えることができた\cite{discord-cloudflare}．

\subsubsection{メッセージ検索基盤}

従来のElasticsearchベースのシステムでは，大量のメッセージ検索において，高負荷がかかってしまうという問題があった．そこで，Kubernetesを使用し，管理しやすいサイズの小さなクラスタ群に分割することで，1つのノードが故障することによる影響範囲を最小限に抑えることができるようになった\cite{discord-indexes-trillions}．一方で，クラスタ分割で耐障害性は確保したものの，依然として個々のコミュニティの巨大化という別の課題があった．すなわち，20億件を超えるような巨大なコミュニティが出現し，このような巨大コミュニティは従来のようにサーバごとにシャードするだけでは，1つのシャードに収まらなくなったのである．そこで，巨大サーバを管理するため，専用のマルチシャード構成を持つインデックスを巨大サーバに対して割り当てた．これによって，Luceneの最大インデックス容量の制限を回避し，並列処理による検索速度の向上を実現した．

\subsubsection{データベース移行：CassandraからScyllaDBへ}

元々Apache Cassandraという分散データベースを使用していた\cite{discord-stores-trillions}．一般的に，ノード数は数十程度であることがほとんどであるが，データ量の増加に伴って最大クラスタが177ノードまで膨れ上がり，パフォーマンス低下や運用コストの増大に直面した．また，Javaで実装されているため，ガベージコレクションによる一時停止が起こり，レイテンシが増加することもあった．そこで，CassandraをScyllaDBに置き換えることにした．ScyllaDBは，Cassandraと互換性のある分散データベースであり，C++で実装されているため，ガベージコレクションによる一時停止がなく，より高いパフォーマンスを提供できる\cite{discord-infoq-scylladb}．さらに，ScyllaDBへの移行段階で，Rustで書かれたデータサービス層を導入することで，クライアントからのリクエストを集約し，ScyllaDBへのアクセスを効率化した\cite{discord-rust-elixir}．結果として，177ノードから72ノードに削減し，突発的な高トラフィックが起こっても，安定したパフォーマンスを維持できるようになった．

\subsubsection{音声通信基盤}

Discordでは，250万人の同時音声接続を支えるためにWebRTCを独自にカスタマイズして使用している．基本的には，クライアント同士が直接通信するP2P方式ではなく，サーバを介して通信する方式を採用している．これによりユーザのIPアドレスを隠すことができるため，セキュリティとプライバシーの向上につながっている\cite{discord-voice-webrtc}．クライアント側でも工夫がなされており，無音声時には帯域とCPU使用率を削減している．他方，バックエンド側では，音声サーバがクラッシュしたり，DDoS攻撃を受けたりした場合に，システムが自動検知し，別の健全なサーバへ瞬時にユーザを移動させることで，耐障害性を実現している．さらに，世界一3リージョン，30以上のデータセンターに850台以上の音声サーバを配置することで，250万人以上の同時接続，220Gbpsを超えるトラフィックを処理できるようにしている．

\section{GitHub}

\subsection{サービス概要}
\begin{itemize}
  \item \textbf{名称：} GitHub
  \item \textbf{トップページURL：} \url{https://github.com}
  \item \textbf{概要：} ソフトウェア開発のためのプラットフォームである．GitHub上にソースコードをホスティングすることで，数百万人もの他の開発者と一緒にコードレビューやプロジェクト管理を行うことができる．
\end{itemize}

\subsection{選んだ理由}
GitHubは，ソフトウェア開発のためのプラットフォームとして，デファクトスタンダードであり，自分自身も日常的に利用している．また，ソフトウェア工学の研究として，普段からGitHub上にホスティングされたプロジェクトのソースコードを解析する機会が多い．そのため，どのような技術的工夫がなされているかに興味を持ち，本サービスを選んだ．

\subsection{定量情報}
GitHubのブログ\cite{github-octoverse-2025}によると，2025年の時点で，GitHub上には1億8000万人以上の開発者が作業を行なっており，平均して，毎秒1人以上の新規開発者がGitHubに参加している．
開発者は毎分230以上の新しいリポジトリを作成し，毎月平均4320万件のPRをマージ，2025年には10億件のコミットをプッシュした．

\subsection{サービスの構成}

本節では，GitHubが大規模化に伴って直面した課題と，それを解決した技術的工夫を，CDN・データベース分割・リポジトリ複製・デプロイ運用の4つの観点に分けて述べる．

\subsubsection{CDN：Fastly}

github.com，GitHub Pages，raw.github.comの静的アセットをFastlyのCDNで配信している\cite{github-fastly}．Fastly導入前はリクエストが急増すると，オリジンサーバが過負荷状態になることがあったが，Fastlyの導入によって個々のPagesのWebサイトの表示速度が大幅に向上し，オリジンサーバへのリクエストが最小限に抑えられた．また，比較によると，Arvixe，Windows Azure，Amazon AWSを凌駕するサイトの読み込み速度を実現している\cite{github-fastly}．

\subsubsection{データベース分割：仮想パーティションとVitess}

次に，アプリケーション層を支えるデータベース基盤について述べる．GitHubの成長に伴い，データベースの拡張において，様々な問題が起こった\cite{github-mysql8}．たとえば，高負荷の環境でレプリカがフリーズし，アプリが応答しなくなるという問題が発生した．そこで，仮想パーティションという概念を導入した．これは，将来的にデータベースを物理的に切り分ける際に問題が発生する可能性があるためである．仮想パーティションは，データベースの論理的な分割を提供し，物理的な切り分けを行う前に，データベースの拡張を可能にする\cite{github-spokes-stretching}．

さらに，仮想的に分離されたスキーマドメインを，物理的に別のデータクラスタへ移動するために，Vitessを使用した\cite{github-partitioning-db}．Vitessの垂直シャーディング機能を利用して，ダウンタイムなしで複数のテーブルをまとめ，本番環境に移行した．具体的には，元となるコピーのデータベースを作成し，元のデータベースを一時的に読み取り専用にし，完全に遅延がゼロの状態を作る．その瞬間にアプリケーションのルーティングを変更し，新しいデータベースへの書き込みを開始する．加えて，さらなる分割のために水平パーティショニングも使用されている．これによりデータベーステーブルを複数のクラスタに分割できるため，より持続可能な成長が可能になる．

\subsubsection{リポジトリ複製：Spokes}

リレーショナルDBの拡張に加え，Gitリポジトリ自体の保存にも独自の工夫がなされている．Spokesは3,800万以上のGitリポジトリと3,600万を超えるgistを保存しているファイルサーバのレプリケーションシステムである．すべてのリポジトリとすべてのgistのコピーを少なくとも3つ保持することで，サーバやネットワークに障害が発生した場合でも永続的で可用性の高いアクセスを提供できる\cite{github-spokes-resilience}．

\subsubsection{デプロイの循環依存とeBPF}

以上のような可用性確保策に加え，GitHub特有のデプロイ運用上の課題として循環依存問題がある．GitHubは自社のソースコードをgithub.com上でホスティングしている．これにより，もしgithub.comがダウンした場合，それを修正するためのコードにアクセスできず，デプロイもできないという循環依存が発生する．GitHubはミラーサーバやバックアップを用意しているが，デプロイ用スクリプトの実行時に予期せぬ依存を持つリスクがあった．たとえば，スクリプトの実行中にGitHubからバイナリをダウンロードする動作，ローカルツールが実行時にアップデート確認のためにGitHubへ通信する，あるいはスクリプトが呼び出した別のサービスがGitHubへ通信するといった，直接的，間接的，推移的な依存が存在することがあった．

このような依存を解決するために，カーネルレベルでフィルタリングする解決策を取った\cite{github-ebpf}．たとえば，ネットワークの通信やDNSクエリを監視し，特定のドメイン（たとえばgithub.com）への通信をブロックすることで，スクリプトの実行中にGitHubへの依存がないことを保証できる．このようにして，GitHubは，github.comへの依存を完全に排除でき，github.comがダウンしても，デプロイ用スクリプトを実行できるようになった．
