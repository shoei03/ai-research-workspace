\section{背景}

\subsection*{破壊的変更と非推奨化の役割}

\begin{itemize}
  \item ライブラリの提供者（API producer）が既存のメソッドを削除・変更するとき、
        消費者（API consumer）のコードはコンパイルエラーや実行時エラーを起こす
        破壊的変更（breaking change）となる
        ~\cite{brito2017java-bc,xavier2017why-break-apis}。
        そのため提供者は削除の前に \texttt{@Deprecated} アノテーションを付与し、
        消費者が代替機能へ移行するまでの猶予を設けることが慣習となっている
        ~\cite{sawant2018why-deprecated}。
  \item \texttt{@Deprecated} を付与する最大の動機は「代替機能の導入（NF）」であり、
        Sawant et al.~\cite{sawant2018why-deprecated} の調査（Java 4 ライブラリ・374 件）で
        全体の \textbf{37\%}（138 件）を占める。
        すなわち、提供者が代替機能 B を追加した時点が
        旧機能 A を非推奨化すべき自然なタイミングとなっている。
\end{itemize}

\subsection*{非推奨化が遅延・放置される構造的問題}

\begin{itemize}
  \item しかし、代替機能が追加された後も旧機能への \texttt{@Deprecated} 付与が
        長期間遅延するケースが広く存在することが、本研究のパイロット分析で示唆される
        （詳細は次節）。
  \item Brito et al.~\cite{brito2017java-bc} が破壊的変更を導入した開発者 56 名に行ったインタビューでは、
        \texttt{@Deprecated} を付与しなかった 17 名への追加質問において、
        \textbf{「非推奨化後のメンテナンスコスト増加（44\%）」}が最多回答であった。
        同論文は「\texttt{@Deprecated} を付与すると削除するまで
        旧版と新版の両方を並行して更新し続けるコストが生じる」ことを理由として挙げており、
        開発者は必要性を認識しつつもコストを理由に後回しにしていることが示されている。
        しかし本研究のパイロット分析では、代替追加から \texttt{@Deprecated} 付与までの
        平均ラグが \textbf{8年}（2,904 日）であることが観察されており（次節）、
        先送りによって並行維持期間が長期化し、
        回避しようとしたコストが逆に増大していることを示唆する。
  \item さらに、Sawant~\cite{sawant2019api-evolution} は
        提供者の非推奨化ポリシー（削除積極性等）が消費者の移行行動を直接左右することを
        大規模分析（25,357 プロジェクト）で実証した。
        提供者が非推奨化を後回しにすることで、
        消費者の移行も連鎖的に遅延するという構造的な問題が生じる。
\end{itemize}

\subsection*{なぜドキュメント非依存の検出が必要か}

現状、先行研究のスコープは付与済みの \texttt{@Deprecated} に限られており、
付与前の放置候補をプロジェクト全体で体系的に把握するツールは存在しない。
技術的負債の議論では、問題が定量的に可視化されることで初めて優先付けの対象となる
\todo{引用が必要（技術的負債の可視化と優先付けに関する先行研究）}。
前節で示した 8 年の遅延を早めるには、「A に代替 B が存在する」ことを提供者に先行して示す必要がある。
しかし根本的な困難がある：\textbf{B が A の代替であるという関係は、
提供者が \texttt{@Deprecated} を付与する時点に初めて宣言されるものである}。
付与前の段階では B が追加されていてもその代替関係はコード上の事実として確定しておらず、
Javadoc に代替が明記されているのはパイロット分析でも \textbf{15.9\%} に過ぎない（詳細は次節）。

先行研究（Sawant~\cite{sawant2018why-deprecated,sawant2018deprecation-needs,sawant2019api-evolution}）は
いずれも付与済みの \texttt{@Deprecated} を起点とするため、
「付与前の候補発見」はスコープ外であった。
したがって、提供者の宣言を待たずに候補を提示するには、
コード構造・使用パターン変化・Git 履歴という客観的シグナルから
\textbf{代替関係を推定する}アプローチが必要となる。
本研究は OSS の Git 履歴を用いた遡及的分析により、
観測可能なシグナルが将来の非推奨化と相関するかを検証する。

\section{アプローチ（提案手法）}

\textbf{ドキュメント非依存の deprecated 候補検出}

\begin{enumerate}
  \item \textbf{動機付け分析}（Javadoc 分析 OK）: Motivation Example の数値で「8年放置が常態」を示す
  \item \textbf{シグナル選定}（ドキュメント禁止）: 以下の特徴量を設計
    \begin{itemize}
      \item 使用パターン変化: caller が method A $\rightarrow$ method B へ段階的移行（Git 履歴）
      \item コード構造類似: 同クラス内での類似シグネチャ・重複パラメータ（AST 解析）
      \item コミット共変: method B 追加コミットで method A の caller が同時変更
      \item 命名パターン: \texttt{getX} / \texttt{getXNew}、\texttt{doY} / \texttt{doYV2} など規則的ペア
    \end{itemize}
  \item \textbf{検出}: 上記特徴量で非推奨化候補を特定するモデル/ルール
  \item \textbf{評価}: 後に @Deprecated が付いたメソッドを、付与\textit{前}の時点で検出できるか（precision / recall）
\end{enumerate}

\section{Research Questions}

\subsection*{動機付け（実証済み、Javadoc 分析）}

代替追加 $\rightarrow$ @Deprecated まで平均 8年（NF先 79\%）$\rightarrow$「今この瞬間にも大量の放置候補がある」

\subsection{RQ1: 提案手法は代替機能をどの程度検出できるか}

\begin{itemize}
  \item 説明変数：入出力が同じか、メソッドが類似しているか\todo{検出手法は要検討}
  \item 検出できたメソッドとできなかったメソッドの違いは何か
  \item 非推奨化直前だと検出しやすいか
  \item スナップショットで類似機能を検索し、
\end{itemize}

\subsection{RQ2: 代替機能を持つメソッドはどのタイミングで非推奨化可能か（学部研究に近い）}

\textbf{Javadoc 等のドキュメントを使わず、コード構造・使用パターン・Git 履歴のみで
「放置 deprecated 候補」を自動検出できるか？}

\begin{itemize}
  \item 代替機能が登場した瞬間に予測できるのが理論値最速
\end{itemize}

\subsubsection*{評価方法}

\begin{itemize}
  \item Ground Truth: deprecated が付与されたメソッド
  \item メソッドを見て、今後8年以内でメソッドが非推奨になるかどうか予測
\end{itemize}
